Where WriteKit fits

Publishing tools split into two camps: hosted editors like Notion or Substack, and static-site generators like Hugo or Astro. Both assume a human clicking through a dashboard or typing in a repo. WriteKit is for the third case — you already spend half your day in an AI chat — and makes your site an operable object from inside that conversation. Pages, collections, publishing: all structured tool calls, no browser-driving.

Not a new CMS. Plain Markdown, standard SQLite, ordinary URLs. The unusual part is the interface — a protocol your AI already speaks.

Why MCP-first

MCP is the open protocol every major AI client already speaks — Claude Code, Cursor, Windsurf, VS Code, OpenCode, Zed. Implement the server once, every client works. Building around it from day one (instead of bolting on an API) gives:

  • OAuth 2.0 + PKCE auth. No API keys; scoped tokens, revocable from settings.
  • Typed tools. create_page, publish_page, reorder_pages — JSON- schema'd function calls, no screen-scraping.
  • Structured context. Read-only resources like writekit://site/drafts let the AI orient itself with no paste-in.
  • Live SSE preview. Side-panel the preview tab; it reloads itself on every edit.
  • Client portability. Swap Claude Code for Cursor next week — same server.

Typical use cases

One command to install, a browser sign-in, a subdomain. Then:

  • Personal blog / essays. Iterate in chat, watch the live preview, publish when it reads right.
  • Docs + changelog. One collection manual-ordered for the docs, one date-ordered for releases.
  • Tutorial series. A collection in learning order; the AI keeps cross-links consistent.
  • Team wiki. Invite editors by email, keep sensitive pages private, share unlisted links for WIP.
  • Research notes. Tag aggressively, let the AI surface related entries when you're writing.
  • Local-first. Desktop build — same tools, same protocol, no account.
Who it's for: people who already write with an AI assistant and want the publishing step to happen in the same conversation — instead of swivelling to a dashboard, pasting Markdown, clicking "publish."

Go backend

WriteKit is a single executable — HTTP server, rendering engine, MCP endpoint, dashboards, migrations, static assets all compiled in via //go:embed. Run it with a config; that's it.

Why Go

Static binaries, stdlib HTTP, goroutines, and cross-compilation to Linux/macOS/Windows — which is what makes the desktop build below feasible. The dashboards are React SPAs built ahead of time and served as embedded static files.

SQLite per subdomain

Each subdomain gets its own SQLite file on disk — pages, collections, versions, settings, all in one file. No shared content database, no row-level tenant column: tenancy is enforced by which file you open.

  • Isolation. Different subdomain = different file with a different handle. One can't read another.
  • Portability. Backup is cp alice.db.
  • Transparency. Standard SQLite — open in any browser to inspect or export.
  • Per-subdomain migrations. No coordinated changes to shared content tables.
  • Content isn't pooled. Postgres only holds platform records — users, tenants, sessions, Stripe IDs. Your pages never sit in a shared table next to anyone else's, which is the whole point: no cross-tenant query can exist, and nothing we run across "all content" is possible by construction. See privacy.
Tradeoff: SQLite is one writer per file. WAL + a connection pool keep readers non-blocking and writes serialised per subdomain — which matches publishing (one person or one AI session writing at a time).

Desktop app

Same codebase, wrapped with Wails. internal/app runs with LOCAL=true, serving the user SPA over loopback in a native window.

What's different in local mode

  • No Postgres, OAuth, Stripe, subdomains, or DNS.
  • Identity is loopback-trust: the server treats the OS user as signed in.
  • Content lives in a single SQLite file in the OS data dir.
Runs offline. The semantic graph uses transformers.js to compute embeddings in-browser — one model download, then your content never leaves the machine.

Open source

github.com/Macawls/writekit — server, desktop, frontends, migrations, MCP tools. Self-hostable via the same docker compose used for dev. No embedding service required; embeddings run client-side.

Data & privacy

What we store, where, and what we don't.

Hosted — on our servers

  • Account: email, OAuth provider, user ID.
  • Subdomain and any custom domain you configured.
  • Pages: in your SQLite file on our hosts.
  • Billing: Stripe customer ID. Card details stay with Stripe.

What we don't collect

  • No analytics, pixels, or session-replay.
  • No training on your drafts.
  • No advertising or data-broker integrations.

Desktop

No data sent to writekit.dev. SQLite in your user data dir. Only outbound traffic is the one-time embedding-model download from Hugging Face if you enable the semantic graph.

Markdown + standard SQLite — export is reading rows from the file. No proprietary format.
